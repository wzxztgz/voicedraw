> 七牛云 × XEngineer 暑期实训 · 题目二：AI 语音绘图工具  
> 版本：1.0 · 最后更新：2026-06

---

## 1. 项目概述

### 1.1 赛题要求

开发一款**纯语音控制**的绘图工具：用户禁止使用鼠标和键盘，全部绘图创作通过语音指令完成。需重点考虑：

1. **指令理解的准确性与容错性**
2. **语音到绘图操作的响应延迟**
3. **复杂指令的拆解与执行**

### 1.2 产品定位

**声绘（VoiceDraw）** 是一款面向桌面浏览器的纯语音智能绘图助手。用户通过麦克风说话，系统实时识别语音、解析意图、渲染图形，并以语音和视觉双通道反馈操作结果。

### 1.3 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端 | 原生 JavaScript (ES Module) + HTML5 Canvas + Vite |
| 后端 | Node.js + ws (WebSocket) |
| 语音识别 | 阿里云 DashScope Paraformer 实时流式 ASR |
| 指令解析 | 规则引擎 + 通义千问 qwen-turbo（双层） |
| 语音合成 | 浏览器 Web Speech API (SpeechSynthesis) |
| 复杂图形生成 | 通义千问 → JSON 配置 → Canvas 渲染器 |

### 1.4 运行环境

- 浏览器：Chrome 100+ / Edge 100+
- 网络：localhost 或 HTTPS（麦克风权限要求）
- 依赖：阿里云百炼 API Key（ASR + LLM 共用）

## 2. 系统架构

### 2.1 整体数据流

```
┌─────────────────────────────────────────────────────────────────┐
│                        浏览器（前端）                            │
│                                                                 │
│  麦克风 ──→ recorder.js ──→ WebSocket ──→ 后端 ASR 代理         │
│                ↑                              │                 │
│                └──────── asr-result ──────────┘                 │
│                          │                                      │
│              ┌───────────┴───────────┐                          │
│              │   isFinal=false       │ isFinal=true              │
│              │   关键词预渲染         │ 指令解析 + 执行           │
│              └───────────┬───────────┘                          │
│                          │                                      │
│              ┌───────────▼───────────┐                          │
│              │   三层指令路由         │                          │
│              │  L1 规则快路径         │                          │
│              │  L2 LLM 结构解析       │                          │
│              │  L3 LLM 图形生成       │                          │
│              └───────────┬───────────┘                          │
│                          │                                      │
│              Canvas 渲染器 ← store 状态管理                       │
│                          │ (并行)                               │
│              语音合成 TTS ← Toast / 任务列表 UI                   │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 三层指令路由（核心架构决策）

| 层级 | 触发条件 | 延迟 | 职责 |
|------|----------|------|------|
| **Layer 1 规则快路径** | 单动作、短句、无复杂信号 | < 50ms | 画圆、改色、移动、撤销等高频指令 |
| **Layer 2 LLM 结构解析** | `hasComplexSignal()` 为 true，或规则返回 unknown | ~1–2s | 复合指令拆句、口语化长句、歧义消解 |
| **Layer 3 LLM 图形生成** | 含图表/流程图/思维导图关键词 | ~2–5s | 生成柱状图、流程图、思维导图等结构化图形 |

**设计原则：** 规则保延迟，LLM 保正确性。不在规则层堆砌正则覆盖所有语言变体，而是用结构信号主动路由到 LLM。

### 2.3 复杂信号检测（Layer 2 入口）

`hasComplexSignal(text)` 在调用规则引擎之前检测，满足任意一条即路由到 LLM：

1. 含复合分隔词：先 / 然后 / 接着 / 最后 / 并且 / 还要 / 之后 / 随后 / 接下来
2. 非句首的「再」作为分隔词（句首「再」交给微调逻辑）
3. 句中 ≥ 2 个绘制动词
4. 句中 ≥ 2 种不同形状词

### 2.4 模块职责

```
client/js/
├── app.js              # 主入口：指令路由、执行引擎、LLM 状态机
├── voice/
│   ├── recorder.js     # 麦克风采集、WebSocket、ASR 句子合并缓冲
│   └── synthesizer.js  # TTS 语音反馈
├── parser/
│   ├── keyword.js      # 规则解析引擎 + 复杂信号检测

├── canvas/
│   ├── renderer.js     # Canvas 渲染调度
│   ├── shapes.js       # 图形定义与绘制
│   ├── grid.js         # 九宫格位置系统
│   ├── charts.js       # 统计图表渲染
```

## 4. 赛题三考点应对方案

### 4.1 指令理解的准确性与容错性

| 问题场景 | 解决方案 |
|----------|----------|
| ASR 误识别同音词 | `HOMOPHONE_MAP` 自动纠正（园→圆、巨形→矩形等） |
| ASR 将长句切成多段 | `recorder.js` 500ms 句子合并缓冲，对上层透明 |
| 规则无法识别的指令 | Layer 2 LLM 兜底解析（`parseBasicCommand`） |
| 规则误判（整句被单条解析抢走） | `hasComplexSignal` 主动路由到 LLM，不等 unknown |
| 绘制意图误触发预览 | `hasDrawIntent` 检测，移动/编辑指令不触发预渲染 |
| 用户说胡话 | `generateClarification` 给建议式反问 |
| 操作可逆 | 撤销/重做（10步）+ 预览确认/取消机制 |

### 4.2 语音到绘图操作的响应延迟

| 路径 | 目标延迟 | 实现手段 |
|------|----------|----------|
| 流式 ASR 中间结果 → 预渲染 | 200–400ms | Paraformer 流式返回 + `detectKeywords` 实时预览 |
| 单条指令规则解析 → 执行 | < 50ms | Layer 1 规则快路径，无网络请求 |
| 单条指令端到端 | < 500ms | 视觉渲染同步，TTS 异步不阻塞 |
| 复杂复合指令 | 1–2s | Layer 2 LLM 解析，显示「正在理解…」提示 |
| LLM 图表生成 | 2–5s | Layer 3 + 描述模式，显示进度 Toast |

**延迟权衡说明：** 简单高频指令走规则保体验；结构复杂的句子主动走 LLM 保正确性。宁可多等 1 秒，也不误画。

### 4.3 复杂指令的拆解与执行

```
用户说：「先画红圆，然后画蓝矩形，最后画红矩形」
         │
         ├─ hasComplexSignal() = true（含先/然后/最后）
         │
         ▼
    LLM 返回 compound JSON
    { type:"compound", tasks:[draw, draw, draw] }
         │
         ▼
    TaskListUI 显示 3 步任务列表
         │
         ▼
    逐步执行，每步间隔 300ms，用户可见进度
         │
         ▼
    语音播报「所有任务已完成」
```

**双路径保障：**

1. **规则路径**：`parseCompound` 按分隔词拆分 → 子句递归 `parseCommand`（延迟低）
2. **LLM 路径**：复杂信号检测 → `parseBasicCommand` 返回 compound（覆盖口语化表达）

## 5. 创新点

### 5.1 可视化预渲染三阶段

用户说话过程中，画布实时显示半透明预览，随关键词逐步推进：

| 阶段 | 条件 | 视觉效果 |
|------|------|----------|
| 1 | 仅有位置/绘制意图 | 红色虚线定位圆 |
| 2 | 颜色已知、形状未确认 | 彩色虚线圆 |
| 3 | 形状已确认 | 半透明实体形状 |

### 5.2 规则 + LLM 三层混合架构

不是「规则 OR LLM」二选一，而是按指令复杂度分层路由，兼顾延迟与正确性。

### 5.3 关联对象同步机制

移动图形时，自动同步：
- 绑定子文字（`_parentId`）跟随平移
- 连线（`_fromId` / `_toId`）端点重新计算
- 删除图形时级联删除关联连线

### 5.4 LLM 图表可继续语音编辑

生成的图表中，数据元素（柱子、节点、数据点）分配独立 ID，用户可在生成后继续说「把3号改成绿色」。

### 5.5 并行反馈设计

视觉渲染（Canvas）同步执行，语音播报（TTS）异步触发，互不阻塞。

## 6. 预渲染机制

```
ASR 中间结果 (isFinal=false)
         │
         ▼
  detectKeywords(text)
         │
    hasDrawIntent? ──否──→ 不触发预览
         │
        是
         ▼
  首次? ──→ _generatePreview()  三阶段预览
  已有? ──→ _updatePreview()    更新参数/升级阶段
         │
  3秒无新输入 ──→ 自动取消预览（防卡死）
         │
  isFinal 到达 ──→ _processCommand() 正式执行 + 清除预览
```

## 7. 已知限制

| 限制 | 说明 | 影响 |
|------|------|------|
| 复杂指令走 LLM 有 1–2s 延迟 | 正确性优先于速度 | Demo 时需说明，界面有「正在理解…」提示 |
| 无自动化测试 | parser 核心逻辑靠人工验证 | 迭代时可能引入回归 |
| 帮助按钮可用鼠标点击 | UI 保留辅助入口 | Demo 全程不使用鼠标即可 |
| 「房子/树/花」关键词未实现渲染 | parseLLMIntent 预留但未扩展 | 用户说「画一棵树」会进入描述模式后失败 |
| 仅支持桌面 Chrome/Edge | 未做移动端适配 | 手机浏览器不可用 |
| WebSocket 无鉴权 | 竞赛 Demo 场景可接受 | 生产环境需补充 |
| 准确率/延迟指标为估算值 | 无自动化基准测试 | 见第 8 节手工测试 |
