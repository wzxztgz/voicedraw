# 1\. 项目概述

## 1\.1 赛题核心要求


开发一款**零鼠标、零键盘**纯语音控制的AI绘图工具，用户所有绘图、编辑、导出、设置操作仅依靠语音指令完成，项目核心考核三大能力：

1. 语音指令理解的**准确性、口语容错性、歧义消歧能力**

2. 语音收音到画布渲染的**分级响应延迟管控**

3. 口语化多步复合指令的**自动拆解、分步有序执行能力**

附加交付要求：梳理项目规划指令能力、最终落地实现能力、裁剪/未完成功能溯源说明，完整复盘架构迭代与问题整改过程。

## 1\.2 产品定位

**声绘（VoiceDraw）**：面向桌面端Chrome/Edge浏览器的轻量化纯语音智能绘图助手。

业务链路：麦克风流式收音 → 阿里云实时ASR语音转文字 → 分层指令路由引擎 → Canvas可视化渲染，产品采用**纯****视觉反馈****交互逻辑**，全程禁用页面鼠标点击操作，严格满足纯语音交互要求。

## 1\.3 产品功能设计核心想法

本项目定位为AI语音绘图工具，全程依托语音指令完成画布操作，结合产品使用局限性、专业场景、对标商用绘图软件，形成完整功能设计思路，具体分为两层设计考量：

1\.产品要实现什么功能

2\.这些功能如何实现。（这部分在2\.系统架构设计和3\.指令能力落地对照表中阐述）

### 1\.3\.1 产品局限性前置研判

语音交互具备天然交互短板：仅依靠口语语音指令操控画布，无法复刻鼠标、键盘精细化拖拽、微调、点对点绘制能力，不适用于手绘、自定义非标图形、像素级精细绘图工作。因此项目开发初期仅落地圆形、矩形、三角形、连线等标准化基础几何图形绘制能力，仅能实现轻量化趣味绘图，但我并不止步于此。

### 1\.3\.2 产品价值定位升级（从趣味工具转向实用工具）

我不愿将本产品打造为仅具备娱乐效果、无实际生产力价值的小众趣味工具，结合自身计算机专业日常学习、办公刚需画图场景：课程流程图绘制、办公思维导图、业务简易统计图绘制，市面上通用绘图工具操作繁琐、鼠标点击步骤多。依托语音免手动操作的优势，本产品能用语音一键快速制图，补齐传统绘图软件手动操作低效痛点，赋予产品实用生产力价值，同时具备公益教学、轻量化办公、无障碍语音绘图的公益价值，后续可拓展轻量化商用绘图能力。

### 1\.3\.3 对标商用软件，做差异化功能设计

为保证产品功能落地性、实用性，本次功能设计参考了主流商用绘图软件【亿图图示】能力体系，做分层适配设计：

1. 对标亿图基础能力：复刻标准化几何图形、连线、编号、配色、删除、移动、修改属性等基础编辑能力，满足基础流程图、架构图搭建；

2. 结合语音交互做差异化精简：舍弃亿图海量非标素材、精细化锚点编辑、图层高阶手绘功能（语音无法适配），聚焦语音可精准操控的标准化制图模块；

3. 补齐语音独有优势能力：新增ASR语音识别、相对点位智能排布、九宫格智能定位、口语语义适配能力，实现纯语音零鼠标快速排版，相比亿图手动拖拽制图，大幅降低制图操作门槛，适配新手、无障碍办公、快速草图场景。

综上，本次所有指令功能、画布架构、解析路由设计，均围绕「参考亿图标准化制图、适配语音交互特性、服务专业办公学习」三大核心落地，而非盲目开发趣味绘图功能。

# 2\. 系统架构设计

## 2\.1 架构全流程迭代复盘


本项目为**3天限时开发交付项目**，指令路由架构依托**每日开发\-晚间****进行系统性的全面****实测复盘\-迭代整改**的流程优化成型，四大架构版本对应三日开发工期，迭代时序清晰、整改链路可追溯：

1. 项目首日完成基础语音绘图指令开发，落地阶段A朴素规则优先架构，首日晚间完成全场景业务测试并复盘架构缺陷；

2. 项目次日迭代优化语义识别能力，落地阶段B复杂信号预判架构，次日晚间复测深挖复合指令、语义分流遗留问题，重构路由逻辑实现阶段C四层路由架构；

3. 项目第三日完成全量解析单元测试、核心代码走查、异常容错补强，最终迭代为完善且健壮的阶段D交付架构，每一轮架构改版均承接前一日实测漏洞，整改落地性极强。

### 阶段A：朴素规则优先架构（初代初稿）


```plaintext
语音收音 → 规则解析parseCommand → 解析成功直接执行 / 识别未知再调用LLM
```

**设计初衷**：此类项目的常规实现链路是 语音输入→前端拾音→ASR服务→文本返回→NLU解析→执行绘图。考虑到语音收音到画布渲染的**延迟**主要分为两个部分：1\.ASR识别（这里采用**流式ASR**）**。**2\.**NLP模型**的调用延迟。于是设计了规则优先架构，高频简单绘图指令依靠正则规则快速解析，降低网络请求延迟；小众复杂指令兜底调用大模型，兼顾效率与通用性。

**实测暴露核心问题**

|业务现象|底层根因|
|---|---|
|「先画圆然后画矩形」仅绘制圆形，后半指令丢失|单绘制指令高置信优先级过高，直接截断整句语义|
|「在右上角画矩形，并和一号连接」仅绘制图形|复合口语分隔词收录不全，无跨动作语义识别能力|
|「把3号改成已完成」误改颜色、误执行操作|改文字、改色、改尺寸指令语义边界耦合混淆|
|「好的画一个圆」直接触发确认指令，绘图失效|确认、帮助类系统指令匹配规则过于宽泛|

**核心复盘结论**：规则解析过于宽松，**误判**以为成功，远比调用LLM时花去的几百毫秒延迟更影响业务，规则不能作为第一优先级执行能力。

### 阶段B：主动复杂信号预判架构（第二版）


新增 `hasComplexSignal()` 复杂语义检测函数，**规则解析执行前**优先甄别多动作复合语句；将复合指令解析前置到图形绘制解析之前，规避单指令抢占语义。

**优化成效**：带有「然后、先、最后」标准连接词的语句，不再被单绘图指令截断拆分。

**遗留短板**：

1. 复杂语义仅判定双绘制动词，连接、改文字跨类型复合语句无法识别


2. 口语常用词「并」未录入复合分隔词白名单（在测试时由这个点发现了框架的问题。）


3. LLM拆解复合指令后，修改文字、导出等指令未做全量映射

**核心复盘结论**：意识到复杂指令的解析陷阱：**如果试图用复杂的正则或规则匹配所有命令，代码很快就会变得无法维护。**考虑到语音指令理解的**准确性、口语容错性、消歧能力**，在规则优先的架构下，根本无法完全利用NLP模型在口语容错性和消除歧义的优势，于是决定全面收紧规则的入口，并加入置信度白名单。（原因：在测试时带入用户视角，意识到口令说多了容易导致嗓子痛，在使用场景上，用户对准确度的要求应该会比延时更高，用户宁愿多等一秒，可能也无法接受识别的错误。需要先撤销上一步操作，然后再执行想执行的任务）于是**将指令理解的准确度的优先级置于延迟前面。**

### 阶段C：四层路由\+置信度白名单架构（第三版定稿架构）


固化四层指令路由机制，重构置信度判定逻辑，核心策略**置信度默认low，仅无歧义指令判定high**，从架构层面杜绝规则误执行。

|路由层级|模块名称|核心职责|
|---|---|---|
|Layer 0|规则复合快路径|标准分隔词拆分语句，子句全高置信，零LLM直接执行|
|Layer 1|复杂语义LLM路径|识别复合口语语句，调用LLM拆解指令，附带规则降级兜底|
|Layer 2|单指令置信度路径|高置信指令直接执行，低置信统一交由LLM兜底|
|Layer 3|大模型图形生成路径|独立链路，生成流程图、思维导图、数据图表|

**核心复盘结论**：指令识别准确度大幅增强。后续继续边界测试，邀请同学做AB测试。

### 阶段D：全维度健壮性补强（最终交付版本）

针对代码健壮性审查、异常容错、工程可维护性、环境兼容性完成最后迭代，新增四大能力补齐架构短板：

1. 新建独立模块 `commandGuard.js`：统一输入清洗、指令参数校验、LLM故障规则降级判定

2. 升级复杂语义判定：双绘制动词升级为**全品类操作动词计数**，覆盖修改、删除、连线全动作

3. 新增parser单元测试文件：27条核心解析用例，支持一键自动化回归测试

4. 新增任务弹窗模块：可视化展示复合指令分步进度，优化用户感知

**项目最终架构核心原则**：规则保障响应低延迟，LLM保障指令正确性；语音交互场景下，指令误执行，远比几百毫秒延迟更不可接受。

## 2\.4 复杂语义判定规则（Layer1准入标准）

函数 `hasComplexSignal(text)`满足**任意一条**，直接准入LLM复杂解析链路：

1. 包含复合分隔词：先/然后/接着/最后/并且/还要/之后，且拆分语句≥2段子指令


2. 文本内识别**≥2个异构操作动词**：绘制、修改、连线、删除、移动、改色等，全覆盖业务动作


3. 文本包含≥2类独立形状名词，内置形状掩码算法，规避「圆角矩形」判定为两个图形的误判

**迭代说明**：动词计数从初代仅统计绘制动词，升级为全业务动词，补齐无连接词复合语句识别能力。

## 2\.5 置信度分流策略（核心规则）


全局置信度判定函数 `_assessConfidence`：**全局默认低置信low，仅标准化无歧义指令准入高置信白名单high**

|指令类型|高置信准入条件|低置信处理逻辑|
|---|---|---|
|modifyText文字修改|语句携带「文字/文本/字体」关键词|无文字关键词，交由LLM消歧语义|
|改色/移动指令|指定目标编号 / 画布已有选中元素|无操作对象，LLM询问\+语义兜底|
|绘制/连线指令|句式完整、参数齐全、指代明确|参数残缺，LLM补齐参数后执行|
|确认/帮助/取消系统指令|纯独立短句，不嵌入业务语句|嵌入绘图语句，禁止抢占业务指令|

# 4\. 赛题三大考点专项解决方案

## 4\.1 考点一：指令理解准确性与口语容错性


针对ASR识别误差、口语歧义、规则误判、大模型服务故障全场景，分层落地容错方案：

|业务风险场景|落地解决方案|架构迭代来源|
|---|---|---|
|方言同音错别字识别|内置同音词映射字典，自动纠错：园→圆、枝→值|阶段A初代优化|
|长语音被ASR截断拆分|收音端500ms延时合并缓冲，拼接碎片化语句|阶段A体验整改|
|规则抢占语义导致误执行|前置复杂语义检测，复杂语句强制隔离走LLM链路|阶段B架构升级|
|高频规则高置信误判|反转置信逻辑，全局默认低置信，严控高置信白名单|阶段C核心重构|
|LLM网络超时、服务宕机|新增指令守卫，合规指令自动回退本地规则执行|阶段D健壮性补强|
|无意义杂音、混乱口语|智能澄清函数，主动弹窗引导用户重新下达指令|阶段A交互优化|
|同音错别字识别|迭代优化prompt，让NLP推测可能的同音词|阶段D健壮性补强|

## 4\.2 考点二：语音到绘图分级延迟管控


采用**分级链路延迟管控**，差异化适配简单/复杂指令，平衡实时性与准确性，主动给用户可视化进度反馈：

|执行链路|目标延迟|延迟优化实现手段|
|---|---|---|
|流式ASR实时预览|200\-400ms|阿里云流式收音，关键词即时匹配，边说边预览|
|Layer0复合/ Layer2高置信|\<500ms|纯前端规则执行，无外网LLM网络请求|
|Layer1口语复合解析|1\-2s|轻量化指令解析Prompt，搭配加载Toast安抚用户感知|
|Layer3专业图形生成|2\-5s|分段渲染\+进度弹窗，拆分图形生成耗时感知|

**核心取舍**：简单标准化指令不走大模型保障极速响应；歧义复杂口语指令，主动舍弃几百毫秒延迟，尽可能保障绘图结果准确。语音交互场景下，用户反复开口纠错消耗的语音成本，远高于短暂等待成本。

## 4\.3 考点三：复杂指令拆解与有序执行


搭建双路径复合指令拆解机制，适配标准口语、自由口语两类复合句式，配套任务进度弹窗提升透明度：

**路径1：Layer0规则快速拆解（标准句式）**

示例：根据中文语义使用分隔词（然后/再/先\.\.\.\.\.\.）拆分子句：例如「先画红圆，然后画蓝矩形」→ 分隔词拆分子指令 → 全部高置信校验 → 本地分步执行 → 任务弹窗分步展示进度，零LLM开销。

**路径2：Layer1大模型智能拆解（自由口语）**

示例：双动词，双形状词或分隔词拆分的子句规则匹配错误则进入LLM：「连接四号和5号，并把五号的文字改成退回」→ 双动词判定复杂语义 → LLM结构化拆分多条指令 → 有序串行执行 → 执行失败自动降级兜底。

**配套交互优化**：新增任务Task弹窗，展示每一步操作文案；执行完成弹窗留存5s；新指令自动清空历史任务，避免指令混淆。

备注：当前复杂指令基本支持“口语化的任意基础指令组合”（除了“画一个矩形然后将它与1号连接”此类指令，根本原因是此类指令依赖线两端的图形标号，而LLM理解时，这个矩形还未生成编号。后续考虑从prompt等方向下手改进。）

# 5\. 项目核心创新点


1. **规则\+LLM四层混合路由（核心架构创新）**：市面上同类工具大多是规则、大模型二选一兜底执行，本项目摒弃这种模式，按照用户指令难易程度分层处理。简单规整语句走零开销规则快通道，自带复杂语义的口语指令，提前识别直接交给大模型处理，不等到解析失败再兜底。项目核心设计理念直白易懂：语音绘图宁可多等一会，也不能画错图形，提前拦截复杂语句，避免简易规则乱解读、误执行指令。

2. **流式三阶段可视化预览（使用体验优化）**：充分利用语音边收音边识别的能力，不用等用户说完一句话，画布就分步给到视觉反馈。第一步识别绘制位置，展示虚线框；第二步识别颜色，切换彩色虚线；第三步确认图形，渲染半透明实体图形。同时增设意图拦截校验，修改、连线类指令不会误弹出预览画面，大幅弱化语音识别等待感，交互感知更流畅。

3. **置信度反转判定（指令解析思路优化）**：和常规工具逻辑完全相反，常规工具只要规则能匹配就直接执行绘图；本项目默认所有口语指令可信度偏低，只有语句完整、语义无歧义的指令，才判定为高可信度直接执行。举例如：「把3号文字改成已完成」表述完整，直接规则执行；「把3号改成已完成」语义模糊，交给大模型分辨意图。优先保证绘图准确率，牺牲少量等待时延，贴合用户语音使用习惯。

4. **双路径复合指令\+任务瀑布流（复杂指令执行）**：适配普通人两种说话习惯拆分多步指令：带「先、然后、并且」连接词的规整话术等，直接靠本地规则快速拆分执行；随口说出、跨操作动作的口语长句，交给大模型拆解分步动作。同时页面弹窗实时展示执行步骤，每一步操作可视化呈现，用户能看清工具正在做什么，不会出现画布突然变动、用户不知情的情况。

5. **独立指令守卫\+故障降级兜底（项目工程优化）**：单独抽离指令校验文件，统一做语音文字清洗、图形编号校验、画布边界拦截，不用零散写在业务代码里。如果大模型接口超时、网络卡顿，合规绘图指令可以自动降级，改用本地规则完成绘制，搭配专属解析测试用例，提前规避语音场景常见报错、边界bug，项目稳定性、可维护性更强。

6. **全场景语音空间指代（交互适配优化）**：搭建完整口语定位体系，适配普通人说话习惯：可以说方位绘图，例如「在右上角画三角形」；可以依托已有图形相对绘图，例如「在1号右边画矩形」；画布图形自动生成数字编号，直接喊编号就能选中、修改、连线图形，全覆盖位置、对象、关联三类口语操控方式，降低语音上手门槛。

7. **画布元素联动绑定（画布一致性创新）**：考虑到实际生产场景，图形、内置文字、连线三者相互绑定。移动图形时，配套文字、连线会跟着一起移动；删除图形时，相连连线一并自动删除；哪怕是大模型一键生成的流程图，节点、文字、边线也全部关联绑定，AI生成的图表，依旧可以语音微调改动，保证整张画布联动统一。

8. **大模型一键出图\+后续语音改图（高阶实用创新）**：直接口述业务需求，就能一键生成流程图、思维导图，区别于市面上AI绘图只能一次性出图的短板。本项目AI生成的所有图形自带编号，生成完毕后，用户可以随时语音改色、改文字、调整布局，实现一键成图、语音微调修改，满足日常学习办公快速制图需求。（大模型prompt持续优化中）

# 6\. 项目后续迭代优化规划

本次项目属于3天限时极速开发交付，受开发时长、算力成本、首期模型选型限制，仍存在用户适配不够细化、模型未择优选型、端云调度不完善、高阶编辑能力缺失等短板。结合全流程实测遗留问题、使用痛点，制定四大长效迭代优化方向，补齐产品能力短板，提升产品实用性与扩展性：

1. **用户需求侧优化迭代**：深入调研在校学生课程制图、职场办公流程图绘制、无障碍人群语音操控三大核心使用人群痛点，分层摸排用户使用偏好，精准划分用户对指令响应延迟、指令识别准确率的敏感等级。后续针对性调整四层路由分流权重、全局置信度判定阈值，适配不同用户使用习惯，优化整体架构适配逻辑，兼顾精准度与响应速度。

2. **多AI大模型横向评测择优选型**：首期开发受3天极短工期限制，为保证项目功能完整落地、接口稳定低延迟，直接选用qwen\-turbo作为底层解析模型，未做多模型对照测评。后续将搭建专属语音绘图标准化测试数据集，围绕指令理解准确率、复合指令拆解成功率、结构化JSON输出稳定性、接口调用延迟、异常容错能力五大核心维度，横向测评qwen\-max、DeepSeek\-V3、GLM\-4多款主流大模型，择优替换底层解析模型，强化复杂口语歧义指令适配能力。

3. **完善端云协同算力调度机制**：优化前端本地规则算力、云端AI算力（或本地ASR/LLM）配比调度逻辑。平衡算力开销、指令响应速度、语义识别精度三者关系，提升前后端联动运行稳定性。

4. **补齐大模型图形增量编辑能力**：当前版本大模型一次性生成流程图、思维导图、业务统计图后，仅支持用户依靠基础语音指令手动修改图形，不支持大模型智能化增量微调。目前基础编辑指令可兜底修改图形，不影响基础办公使用；后续新增画布图形上下文记忆模块，留存图表结构、编号、样式上下文信息，支持用户语音下达微调指令，直接调用大模型增量修改已生成专业图表，省去分步手动修改流程，拔高高阶制图编辑效率。
